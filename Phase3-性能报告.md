## 二、SQLite 存储增强 (v0.1.1)

### 2.1 连接池架构

```
┌─────────────────────────────────────────┐
│              SQLiteStorage               │
├─────────────────────────────────────────┤
│  写连接 (thread-local, 独占)             │
│  ├─ WAL mode                            │
│  ├─ 8MB page cache                      │
│  ├─ 256MB mmap                          │
│  └─ NORMAL synchronous                  │
├─────────────────────────────────────────┤
│  读连接池 (thread-safe, 可配置大小)       │
│  ├─ 只读 URI 模式                       │
│  ├─ LRU 回收 (300s 空闲超时)            │
│  └─ 池耗尽时回退到写连接                 │
└─────────────────────────────────────────┘
```

### 2.2 新增方法

| 方法 | 说明 |
|------|------|
| `incremental_vacuum(pages=100)` | 渐进式空间回收（非阻塞） |
| `optimize()` | 数据库优化：ANALYZE + FTS5 optimize + WAL checkpoint |
| `db_size()` / `db_size_mb()` | 数据库文件大小监控 |
| `analyze()` | 更新查询计划统计 |
| `checkpoint(mode)` | WAL 检查点控制 |
| `auto_vacuum_check()` | 自动阈值触发回收 |

### 2.3 性能 PRAGMAs

```sql
PRAGMA cache_size=-8000       -- 8MB 页面缓存
PRAGMA mmap_size=268435456    -- 256MB 内存映射
PRAGMA synchronous=NORMAL     -- WAL 模式下安全降级
PRAGMA temp_store=MEMORY      -- 临时表存内存
PRAGMA busy_timeout=5000      -- 5s 忙等待
```

### 2.4 构造函数增强

```python
SQLiteStorage(
    db_path="./memvault.db",
    pool_size=3,              # 读连接池大小（0=禁用）
    auto_vacuum=False,        # prune 后自动回收
    vacuum_threshold_mb=50.0, # 触发 auto_vacuum 的阈值
)
```

## 三、基准测试结果

### 3.1 批量写入对比

| 条目数 | MemoryStorage | SQLiteStorage | SQLite Batch | 加速比 |
|--------|--------------|---------------|-------------|--------|
| 100 | ~2ms | ~8ms | ~5ms | 1.6x |
| 1,000 | ~15ms | ~45ms | ~25ms | 1.8x |
| 5,000 | ~65ms | ~200ms | ~110ms | 1.8x |

**结论**: 
- MemoryStorage 在 < 10K 条目时快于 SQLite（无磁盘 I/O）
- SQLite `add_batch()` 比逐条 INSERT 快 1.6-1.8x（单事务批量提交）
- SQLite 的优势在 10K+ 条目时显现（增量更新 + 磁盘持久化）

### 3.2 读取性能

| 操作 | MemoryStorage | SQLiteStorage | 说明 |
|------|--------------|---------------|------|
| 顺序读 500 条 | ~0.3ms | ~1.5ms | Memory 更快（纯 dict） |
| 随机访问 50 条 | ~0.1ms | ~0.8ms | 两者都很快 |
| 字符计数 | ~0.05ms | ~0.3ms | SQL 聚合查询 |

**结论**: Memory 在读取速度上始终领先，但 SQLite 提供了持久化能力。

### 3.3 搜索性能

| 搜索方式 | MemoryStorage | SQLiteStorage | 加速比 |
|----------|--------------|---------------|--------|
| 子串匹配 (Python) | ~1.2ms | ~0.8ms | 1.5x |
| FTS5 全文搜索 | N/A | ~0.3ms | **4x** vs Python |

**结论**: FTS5 全文搜索比 Python 子串匹配快 4 倍，且 SQLite 原生支持。

### 3.4 元数据更新

| 操作 | MemoryStorage | SQLiteStorage (batch) |
|------|--------------|----------------------|
| 50 条逐条更新 | ~0.8ms | ~3ms |
| 100 条批量更新 | ~0.3ms | ~1.5ms |

**结论**: 批量更新在两种后端的性能差异不大，瓶颈在锁粒度。

## 四、百万 Token 压力测试

### 4.1 测试配置

| 参数 | 值 |
|------|-----|
| 字符上限 | 50,000 chars (~20,000 tokens) |
| 半衰期 | 7 天 |
| 权重区间 | [0.3, 3.0] |
| 周期数 | 30 |
| 嵌入模型 | MockEmbedder (32-dim) |
| 存储后端 | MemoryStorage |

### 4.2 检索指标说明

> ⚠️ **v0.1.2 修订**: 旧版报告的"100% 命中率"存在循环论证（关键词注入 → 子串匹配 → 非空=命中）。
> v0.1.2 升级了压力测试的检索路径：
> - 嵌入器：`TopicMockEmbedder` (128-dim, 主题感知) 替代 `MockEmbedder` (纯随机)
> - Dense：`SimpleDenseRetriever` (in-memory cosine, 零外部依赖) 替代仅 LlamaIndex
> - 三路全开：`dense=True, sparse=True, colbert=True`，源标签 `D+S+C`
> - 全局 `relevance_threshold=0.35` 过滤低分对抗查询

### 4.3 三场景汇总

| 场景 | 累积 Token | 保留 Token | 有效 Token | 裁剪率 | P@5 | MRR | 对抗空 | 总耗时 |
|------|-----------|-----------|-----------|--------|------|------|--------|--------|
| **A: 均匀流** | 999,984 | 19,978 | 19,978 | 98.0% | 1.00 | 1.00 | **64.0%** | 51,616ms |
| **B: 突发流** | 1,053,124 | 19,944 | 19,944 | 98.1% | 1.00 | 1.00 | **45.3%** | 50,012ms |
| **C: 长文流** | 999,984 | 19,918 | 19,918 | 98.0% | 1.00 | 1.00 | **26.7%** | **32,431ms** |

### 4.4 检索质量分析

#### 正例查询（Precision@5 = 1.00, MRR = 1.00）

三路混合检索 (Dense + Sparse + ColBERT) 对关键词注入文本的检索精度极高。
- **Dense**: TopicMockEmbedder 按主题质心分组，同主题余弦相似度 ~0.96
- **Sparse**: BM25 bigram 捕获关键词匹配
- **ColBERT**: 字符级 MaxSim 重排序
- **RRF 融合**: 三路信号叠加，正例查询得分远高于对抗查询

#### 对抗查询（空结果率 26.7% ~ 64.0%，大幅提升）
对抗查询（如"GraphQL查询优化"、"Rust内存安全"等不在文本池中的主题）**大部分被正确拒绝**。

- **均匀流 (64.0%)**: 条目多、文本多样 → BM25 IDF 区分度高 → Dense 低分 + Sparse 弱匹配 → 过滤
- **突发流 (45.3%)**: 文本量波动大 → 部分周期 IDF 分布不均 → 过滤率稍低
- **长文流 (26.7%)**: 条目少 → BM25 IDF 区分度低 → 更多 bigram 重叠通过

**未达到 100% 的原因**: TopicMockEmbedder 仅模拟 5 个主题，真实 bge-m3 的 1024-dim
语义空间能提供更强的区分度。对抗查询中如"优化"等字仍在文本池其他上下文出现，
Sparse/ColBERT 路径会产生弱匹配。生产环境使用 bge-m3 后预期可达 80%+。

完整的三路混合检索质量测试（含真实 bge-m3 dense + ColBERT MaxSim）参见 `tests/test_sleep_loop_quality.py`。

### 4.5 场景 A: 均匀流 — 详细采样

| 周期 | 累积 Token | 保留 Token | 裁剪率 | 命中 | P@5 | MRR | 条目 | 耗时 |
|------|-----------|-----------|--------|------|------|------|------|------|
| 1 | 33,333 | 19,999 | 40.0% | 5/5 | 1.00 | 1.00 | 267 | 1,150ms |
| 5 | 166,666 | 19,971 | 88.0% | 5/5 | 1.00 | 1.00 | 269 | 1,317ms |
| 10 | 333,332 | 19,962 | 94.0% | 5/5 | 1.00 | 1.00 | 270 | 1,301ms |
| 15 | 500,000 | 19,968 | 96.0% | 5/5 | 1.00 | 1.00 | 270 | 1,321ms |
| 20 | 666,664 | 19,998 | 97.0% | 5/5 | 1.00 | 1.00 | 270 | 1,442ms |
| 25 | 833,332 | 19,952 | 97.6% | 5/5 | 1.00 | 1.00 | 270 | 1,323ms |
| 30 | 999,984 | 19,978 | 98.0% | 5/5 | 1.00 | 1.00 | 270 | 1,357ms |

**分析**: 
- 裁剪率从 40% 逐步上升到 98%，稳定收敛
- 保留 token 始终在 ~20K（char_limit 50K / 2.5）
- P@5 始终 1.00（BM25 准确找到注入的关键词）
- MRR 始终 1.00（第一个结果总是相关）
- 条目数稳定在 ~270 条
- 每周期耗时 ~1300ms（含稀疏索引重建 ~100ms）

### 4.6 场景 B: 突发流 — 应对随机负载

随机范围 5K~60K tokens/cycle，模拟真实场景的突发性。

**关键发现**:
- 最大单周期注入: 147,227 chars (~58,890 tokens) → 周期 30
- 该周期耗时: 2,750ms（正常范围）
- 即使突发注入 60K tokens，系统在 3 秒内完成
- 裁剪率始终在捕获后收敛到 98%

### 4.7 场景 C: 长文流 — 最优性能

长文本偏重意味着更少的条目数 → 更少的 `add()` 和 `rebuild_index()` 调用。

**性能对比**:
| 指标 | 均匀流 (A) | 长文流 (C) | 提升 |
|------|-----------|-----------|------|
| 总耗时 | 39,461ms | 4,975ms | **7.9x** |
| 每周期平均 | 1,315ms | 166ms | **7.9x** |
| 添加耗时占比 | 94% | 43% | — |

**结论**: 条目数是性能的主要决定因素。长文本（少条目）场景下，百万 token 在 **5 秒**内完成。
